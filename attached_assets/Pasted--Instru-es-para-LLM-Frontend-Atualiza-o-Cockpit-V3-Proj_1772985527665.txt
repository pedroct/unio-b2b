## Tarefa 1: Ativar Card do Score Funcional

### O que mudar

O 4º card do Cockpit ("Score Funcional") atualmente está hardcoded como bloqueado. A API agora retorna `"ativo": true` para o tipo `"functional"`.

### Lógica de renderização

```
SE score.tipo == "functional" E score.ativo == true:
  SE score.score != null:
    → Renderizar card ATIVO com:
      - Valor do score (ex: 78.5)
      - Badge de classificação (score.classificacao)
      - Componentes (se score.componentes existir)
  SENÃO:
    → Renderizar card com mensagem: score.mensagem_bloqueio
      Ex: "Coleta em andamento. Faltam biomarcadores funcionais."
```

### JSON de exemplo — Score Funcional ativo

```json
{
  "tipo": "functional",
  "ativo": true,
  "score": 78.5,
  "classificacao": "good",
  "tendencia_score": "estavel",
  "is_partial": false,
  "mensagem_bloqueio": null,
  "componentes": {
    "velocidade_caminhada": {
      "valor": 1.32,
      "valor_formatado": null,
      "unidade": "m/s",
      "referencia": "Média 30d"
    },
    "forca": null,
    "volume_treino": {
      "valor": 210.0,
      "valor_formatado": null,
      "unidade": "min/semana",
      "referencia": "Soma 7d"
    },
    "estabilidade": {
      "valor": 0.82,
      "valor_formatado": null,
      "unidade": "%",
      "referencia": "Walking Steadiness"
    }
  }
}
```

### JSON de exemplo — Score Funcional sem dados (coleta em andamento)

```json
{
  "tipo": "functional",
  "ativo": true,
  "score": null,
  "classificacao": null,
  "mensagem_bloqueio": "Coleta em andamento. Faltam biomarcadores funcionais (velocidade, treino, estabilidade)."
}
```

### Seção de Componentes Funcionais (se score.componentes existir)

Renderizar seção "Componentes — Score Funcional" com até 4 cards:

| Campo API                          | Label no Card        | Unidade    | Observação                             |
| ---------------------------------- | -------------------- | ---------- | -------------------------------------- |
| `componentes.velocidade_caminhada` | Velocidade Caminhada | m/s        | —                                      |
| `componentes.forca`                | Força                | nível      | **Pode ser `null`** (métrica opcional) |
| `componentes.volume_treino`        | Volume de Treino     | min/semana | —                                      |
| `componentes.estabilidade`         | Estabilidade         | %          | Walking Steadiness                     |

**Se `componentes.forca` for `null`**, não renderizar o card de Força.

**Se `is_partial == true`**, exibir um indicador visual (badge ou tooltip) informando "Score parcial — métrica ausente".

## Tarefa 2: Corrigir Exibição de Duração de Sono

### O que mudar

Os componentes `sono_total` e `sono_rem_profundo` do Score Recuperação agora incluem um campo novo: `valor_formatado`.

### Antes (❌)

```
Sono Total:     327.3 min
REM + Profundo: 147.35 min
```

### Depois (✅)

```
Sono Total:     5h 27min
REM + Profundo: 2h 27min
```

### JSON atualizado

```json
{
  "sono_total": {
    "valor": 327.3,
    "valor_formatado": "5h 27min",
    "unidade": "h",
    "referencia": "Média 7d"
  },
  "sono_rem_profundo": {
    "valor": 147.35,
    "valor_formatado": "2h 27min",
    "unidade": "h",
    "referencia": "Média 7d"
  }
}
```

### Regra de renderização

```
SE componente.valor_formatado existir E não for null:
  → Exibir valor_formatado como texto principal do card
SENÃO:
  → Exibir componente.valor + " " + componente.unidade (fallback)
```

**Aplicar esta regra em TODOS os componentes de TODOS os scores**, não apenas no sono. O campo `valor_formatado` é genérico.

## Tarefa 3: Atualizar Régua de Classificação

### O que mudar

A régua de classificação (cores/badges) mudou de **90/75/60** para **80/60/40**. O backend já retorna a classificação correta no campo `classificacao`, mas se o frontend tiver lógica local de cores baseada nos thresholds antigos, ela precisa ser atualizada.

### Antes (❌)

| Range  | Classificação |
| ------ | ------------- |
| 90–100 | Excelente     |
| 75–89  | Bom           |
| 60–74  | Atenção       |
| 0–59   | Risco         |

### Depois (✅)

| Range  | Classificação | Campo `classificacao` | Cor sugerida |
| ------ | ------------- | --------------------- | ------------ |
| 80–100 | Excelente     | `"excellent"`         | 🟢 Verde     |
| 60–79  | Bom           | `"good"`              | 🔵 Azul      |
| 40–59  | Atenção       | `"attention"`         | 🟡 Amarelo   |
| 0–39   | Risco         | `"risk"`              | 🔴 Vermelho  |

### Impacto

O **Score Cardiovascular de 90** que atualmente é mostrado como "Bom" (threshold antigo de 90) deveria ser **"Excelente"** (novo threshold de 80). O badge de classificação deve usar o valor retornado pelo campo `classificacao` da API, e NÃO aplicar lógica local.

### Recomendação

```javascript
// ❌ NÃO FAZER — lógica local de classificação (desatualizada)
function getClassificacao(score) {
  if (score >= 90) return "excelente";
  // ...
}

// ✅ FAZER — usar o valor da API
const classificacao = score.classificacao; // "excellent", "good", "attention", "risk"
```

**Isso afeta TODOS os 4 scores** (Cardiovascular, Metabólico, Recuperação, Funcional).

## Tarefa 4: Score Funcional no Gráfico de Tendência

### O que mudar

O endpoint `/historico-scores` agora retorna o campo `funcional` em cada ponto do histórico.

### JSON

```json
{
  "data": "2026-03-08",
  "cardiovascular": 90.0,
  "metabolico": null,
  "recuperacao": 76.0,
  "funcional": 78.5
}
```

### Regra

- Adicionar nova série "Funcional" no gráfico de tendência
- Usar cor distinta (sugestão: roxo `#8B5CF6` ou laranja `#F97316`)
- Se `funcional == null`, não plotar ponto naquele dia
