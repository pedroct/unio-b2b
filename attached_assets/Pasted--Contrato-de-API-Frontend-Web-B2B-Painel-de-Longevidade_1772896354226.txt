# Contrato de API - Frontend Web (B2B) - Painel de Longevidade V2

Este documento estabelece o contrato de comunicação entre a aplicação Web Frontend (Painel B2B para Profissionais) e o Backend, englobando a versão 2.0 do motor de pontuação.

A V2 desbloqueia dois novos pilares matemáticos na resposta: o **Score Metabólico (`metabolic`)** e o **Score de Recuperação (`recovery`)**, preenchendo novos nós no objeto dinâmico de `componentes`.

---

## 1. Visão Geral do Cockpit (Resumo dos 4 Estruturais)

**Endpoint:** `GET /api/longevidade/clientes/{cliente_id}/cockpit`
**Autenticação:** Requer Bearer Token (Profissional)

O schema do Cockpit foi expandido. Os blocos `metabolic` e `recovery` que antes retornavam sempre com `ativo: false` (Bloqueado) agora podem vir com `ativo: true` caso o paciente tenha métricas suficientes para calculá-los.

Se `ativo: true`, as chaves `score`, `classificacao` e `componentes` estarão populadas com dados.

### Dicionário de Classificações

A interface do frontend deve renderizar esquemas de cores de acordo com a enumeração estrita das classificações:

- `"excellent"` (Excelente - Verde escuro/Premium)
- `"good"` (Bom - Verde claro)
- `"attention"` (Atenção - Amarelo/Laranja)
- `"risk"` (Risco Aumentado - Vermelho)

## 2. Histórico de Scores (Gráficos de Tendência)

**Endpoint:** `GET /api/longevidade/clientes/{cliente_id}/historico-scores`
**Parâmetros de Query:** `?intervalo=30d` (Suporta "30d", "90d", "365d")
**Autenticação:** Requer Bearer Token

A estrutura de dados permanece uma matriz focada em _Time Series Graphic Plotting_. Na V2, as chaves `metabolico` e `recuperacao` passam a ser populadas com valores float em tempo real, lado a lado do `cardiovascular`.

### Exemplo de Response (Status 200 OK)

```json

  "cliente_id": 1234,
  "historico": [

      "data": "2026-03-01",
      "cardiovascular": 85.5,
```

## 3. Comportamento Semântico UI/UX (Fallback)

A ausência do `Apple Watch` (Recuperação) ou de métricas de Bioimpedância (Gordura corporal) afetará um determinado scorecard.

Quando houver insuficiência de dados primários daquele paciente para rodar qualquer motor na diária, a resposta do backend fará fallback de `ativo: true` para `ativo: false`, contendo uma mensagem de falha em `mensagem_bloqueio`.

Exemplo:

```json
{
  "tipo": "metabolic",
  "ativo": false,
  "mensagem_bloqueio": "Coleta em andamento. Faltam biomarcadores metabólicos (Ex: Peso, Cintura).",
  "componentes": null
}
```

_O Frontend deverá substituir o modal clicável pela mesma caixa opaca de visualização da v1._
